多 Agent 协作，也许只是一场热闹的菜市场

2026-06-25 20:00·阅读时间：6 分钟

最近一年多，"Multi-Agent" 这个词在 AI 圈里热得发烫。几乎所有做 Agent 的框架——LangGraph、CrewAI、AutoGen、OpenAI Swarm——都在主打"多 Agent 协作"。宣传图上一个 Agent 跟另一个 Agent 连线、对话、分工，看起来就像一个小型数字化公司。架构图画得密密麻麻，节点和箭头眼花缭乱，仿佛只要 Agent 够多，问题就会自动解决。

但我越看越觉得不对。

Agent-Graph：一场互相对话的菜市场

所谓 Agent-Graph，本质上就是把多个 Agent 连成一张图，你调用我、我调用你，互相传递上下文。每个 Agent 各司其职——一个负责搜索、一个负责写代码、一个负责审查、一个负责总结。它们之间通过消息传递来"协作"，最后产出一个结果。

听起来很美。但实际操作起来是什么感觉呢？

就像你去菜市场买菜。摊主 A 喊一句"我这有最好的西红柿"，摊主 B 回应"他那不行，来我这买黄瓜"，旁边的摊主 C 插嘴"你们都别吵了，听我的"。每一个 Agent 都在说话，但没有人真正在听。上下文在 Agent 之间传来传去，每传一次就丢失一点信息、多一层误解。到最后，产出的结果往往不如一个专心干活的单 Agent。

这还不是最糟的。最糟的是，Agent-Graph 给你一种"看起来在做很多事情"的错觉。日志刷刷地刷屏，每个 Agent 都在"思考"、都在"输出"，热热闹闹十几轮对话过去了——然后你一看结果，跟一个配置好 tools 的单 Agent 跑两轮的效果差不多，甚至还不如。

我最近试了一些 multi-agent 框架写 SWE 任务。一个 Agent 负责读代码、一个负责定位问题、一个负责写 patch、一个负责 review。四个 Agent 来回对话了十几轮，token 烧了一大把，最后产出的 patch 跟 claude-code 单 Agent 一键生成的比起来——质量更低、花费更多、时间更长。

问题是：我为什么要为更差的结果付更多的钱？

树形 Multi-Agent：也许有一点点用？

相比之下，树形结构的多 Agent 可能稍微合理一点。

所谓树形，就是一个"主 Agent"负责任务分解和调度，然后把子任务分发给不同的 worker Agent，worker 之间互不通信，只向主 Agent 汇报结果。这个结构至少避免了菜市场式的"你一句我一句"——Agent 之间没有横向通信，信息流是单向的、可控的。

我能想到的合理场景是大规模项目迁移。比如把一个十万行的项目从 Vue 2 迁移到 Vue 3，你可以把不同模块分给不同的 Agent 并行处理，最后再汇总。这种场景下，任务天然可拆分，Agent 之间不需要协作，各自独立工作就行。

但说实话，我作为一个学生，目前还没有真正遇到过这种场景。而且就算是这种场景，核心难点也不在 multi-agent 的"协作"，而在于任务拆分的质量和汇总时的冲突解决。这两个问题，目前没有一个 multi-agent 框架能自动做好。

我们到底在优化什么？

回过头来看整个 multi-agent 热潮，我觉得有一个根本问题没有被认真回答：我们到底在优化什么？

如果目标是"让 AI 写出更好的代码"，那 multi-agent 并没有提供令人信服的证据。Claude Code 的单 Agent + 工具调用模式，在绝大多数 SWE 任务上表现碾压各种多 Agent 框架。OpenAI 的 Codex CLI、Aider、Cline——这些最实用的编码工具，全部都是单 Agent 架构。

如果目标是"更低的成本"，那就更不对了。多 Agent 来回对话，token 消耗是单 Agent 的数倍甚至数十倍。每一轮 Agent 间的上下文传递都是在给 API 账单加码。

如果目标是"更可靠的输出"——多个 Agent 互相检查、互相纠错——理论上说得通，但实践中 Agent 的"幻觉"是会传染的。一个 Agent 的错误输出被另一个 Agent 当作事实接受，然后基于这个错误继续推理。你加的不是纠错能力，而是错误的传播链。

Plan 和 TODO 才是真正重要的

在我为数不多的动手经验里，真正提升 Agent 产出质量的，从来不是加更多 Agent，而是两件极其朴素的事：

第一，Plan。 在动手之前，先让 Agent 做一个详细的计划。不是那种"我会分析代码然后修复问题"的敷衍计划，而是具体到"我会先读 A 文件理解数据结构，然后检查 B 文件的第 50-80 行，接着修改 C 函数的参数签名，最后更新对应的测试"。当 Agent 被迫把思考过程显式化、结构化之后，产出的代码质量有肉眼可见的提升。

第二，TODO。 把大任务拆成一系列小的、可验证的步骤，每完成一步就打一个勾。这看起来笨拙，但极其有效。它强制 Agent（或者你自己）在每个阶段停下来确认方向正确，而不是一口气冲到底然后发现从一开始就跑偏了。

这两个东西，跟 multi-agent 一点关系都没有。它们是工程方法，不是架构花样。

也许是我太年轻

我当然承认，我的视角是有限的。一个还在读书的学生，没有经历过真正复杂的企业级项目，没有管理过几十个工程师协作的代码库，也没有遇到过那种"一个 Agent 的 context window 根本装不下"的极端场景。

也许在某些领域——比如多模态协作（一个 Agent 看图片、一个 Agent 读文本、一个 Agent 调 API）、或者超长周期的自主研究任务——multi-agent 确实有它的用武之地。也许当任务复杂到单个 context window 无论如何也塞不下的时候，拆分到多个 Agent 是一种必然，而不是选择。

但在 SWE 领域，在我目前能接触到的所有场景里，multi-agent 更像是一个被过度营销的概念，而不是解决实际问题的工具。它画出来的架构图好看，讲出来的故事动听，但等你真的跑起来——就是一场热热闹闹的菜市场。

写在最后

Claude Code 最近的工作流（workflow）功能其实也是一种多 Agent，但它有一个关键的不同：Agent 之间不对话。每个 Agent 独立工作、独立返回结果，由脚本来编排和汇总。这避免了"传话游戏"式的信息衰减，也避免了"菜市场"式的混乱交互。我觉得这个方向是对的——不是"让 Agent 像人一样开会"，而是"让 Agent 像工具一样被编排"。

说到底，软件工程里的一个古老智慧在这里同样适用：简单的东西容易做对，复杂的东西容易做错。 在你拿起 multi-agent 这把花里胡哨的瑞士军刀之前，先问问自己——一个 Agent，配上清晰的 Plan 和踏实的 TODO，是不是就已经足够好了？

大概率，是的。
