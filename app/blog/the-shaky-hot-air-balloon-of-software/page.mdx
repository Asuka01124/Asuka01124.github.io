# 摇摇欲坠的热气球：从 Agent 开发看软件工程的复杂性

<PostMeta date="2026-06-14T23:30:00" readingTime={5} />

最近几个月，我一头扎进了 AI Agent 的世界。Tools、Skills、Context Engineering——这些概念一下子大量涌入有种被唬住的感觉，实际操作却也不是无法理解。一个简单的 Agent Loop，核心逻辑不过几十行代码：接收输入 → 调用大模型 → 解析工具调用 → 执行 → 把结果喂回去 → 循环。在一个 Python 脚本里就能跑起来，丝滑得不行。

然后我也想自己真正设计一个能用的mini的swe agent，我开始读 [opencode](https://github.com/sst/opencode) 的源码。

---

## 几十行 vs 几万行

那个"几十行的 Agent Loop"，在 opencode 里变成了什么？Tool 注册与发现系统、Permissions 权限校验层、Context Window 的 token 预算管理、流式输出的 buffer 处理、各种异常重试、超时控制、并发工具调用的编排、不同模型 provider 的适配、TUI 的渲染与交互……随便一个模块就上千行。

一开始我很困惑：这些不能简化掉吗？一个 Agent 不就是调 API 吗？

后来我慢慢懂了。那几十行代码，在我的电脑上、我的网络下、我熟悉的输入下，确实没有任何问题。但真实世界里，每个用户的环境都不一样，每个边界情况都是一个潜在的崩溃点。那些"多余"的代码，不过是在替所有可能出错的地方提前挡一刀。

---

## 软件工程最难的，从来不是"让它跑起来"

这是我最近反复想到的一个观点。任何一个计算机系的学生都可以在几天内搭一个能用的 Web 应用。但能用，和能被真实的人稳定地用，是两件截然不同的事。

我在课堂上写的代码，输入永远是合法的，网络永远是通的，环境永远是我自己配置好的。但一旦软件交到真实用户手里，这些前提就全部失效了。用户会做你没预料到的事，遇到你没测试过的情况，在你不了解的环境里运行你的程序。

软件工程的复杂度，很大一部分就藏在这里：不是让逻辑跑通，而是让逻辑在所有情况下都能优雅地跑通，或者优雅地失败。

---

## 臃肿是必然的，屎山是结构性的

于是项目开始膨胀。任何大型开源项目，如果你觉得它"太复杂了""屎山一样"，大概率不是作者水平不行，而是它替你扛下了这个世界所有的脏活。

那些"臃肿"的代码——错误处理、边界检查、兼容层、向后兼容的遗留接口——才是软件真正在"服务人"的部分。核心逻辑是大脑，而这些是免疫系统。没有免疫系统的人活不过一个流感季节，没有这些代码的软件活不过第一个真实用户。

读 opencode 之前，我觉得写好代码就是把逻辑写清楚。读完之后，我意识到大量的代码其实是在和"这个世界不按你预期运转"这件事做斗争。

---

## 摇摇欲坠，但飞着

网上有个说法我特别喜欢：**"现代计算机世界是一个摇摇欲坠的热气球，靠着无数人的修补和祈祷维持在空中。"**

我们每天都在上面加补丁。一个开源项目依赖另一个开源项目，那个又依赖另外十个。随便哪一个环节出问题——一个维护者 burnout 了、一个关键包被恶意篡改了、一个 API 不兼容更新了——整个链条都可能断裂。

但它没有断。至少大部分时候没有。

这个摇摇欲坠的热气球还在飞。数亿人每天在用它通信、工作、娱乐。这本身就是一个奇迹。而软件开发者的工作，本质上就是在这个气球上找到漏洞，缝上一针，再在附近加个备用的补丁。

---

## 那还要不要写 code？

当然要。正因为开始理解这些复杂性，反而更欣赏那些把复杂包装成简单的项目。好的工具就该像一把锋利的刀——你握住刀柄时感觉不到刀柄内部有多少层锻造工艺。

我还是个大二的学生，很多工程上的弯路还没亲自走过。但我想先在脑子里装下这件事：真正的软件工程，不只是让代码跑起来，而是让它在你看不见的地方也能撑住。

---

*写于阅读 opencode 源码后的某个深夜。世界是个摇摇欲坠的热气球，但我们还在上面。2026年6月14号*
